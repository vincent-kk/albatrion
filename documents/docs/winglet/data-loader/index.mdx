<PackageHeader name="@winglet/data-loader" />

Batching and caching utility for asynchronous data fetching, inspired by GraphQL DataLoader.
Automatically collects individual `load()` calls within the same tick into a single
batch request, eliminating the N+1 query problem with a built-in promise cache.

## Installation

```bash
yarn add @winglet/data-loader
```

## API

### Constructor

```ts
import { DataLoader } from '@winglet/data-loader';

const loader = new DataLoader<Key, Value>(batchLoader, options?);
```

`batchLoader` receives an array of keys and must return a `Promise` resolving to values (or `Error`) in the same order.

### Instance Methods

| Method | Description |
|---|---|
| `load(key)` | Returns `Promise<Value>` for a single key; batches automatically |
| `loadMany(keys)` | Returns `Promise<(Value \| Error)[]>` for multiple keys |
| `clear(key)` | Removes a single key from the cache |
| `clearAll()` | Clears the entire cache |
| `prime(key, value)` | Pre-populates the cache |

### Options

| Option | Type | Default | Description |
|---|---|---|---|
| `batch` | `boolean` | `true` | Enable request batching |
| `maxBatchSize` | `number` | `Infinity` | Max keys per batch |
| `cache` | `boolean` | `true` | Enable promise caching |
| `cacheKeyFn` | `(key) => CacheKey` | identity | Map key to cache key |
| `cacheMap` | `MapLike` | `new Map()` | Custom cache implementation |
| `scheduler` | `(fn) => void` | `process.nextTick` | Controls batching timing |

## Usage Examples

```ts
import { DataLoader } from '@winglet/data-loader';

const userLoader = new DataLoader<number, User>(async (ids) => {
  const users = await fetchUsersByIds(ids);
  return ids.map(id => users.find(u => u.id === id) ?? new Error(`Not found: ${id}`));
});

// These three calls batch into a single fetchUsersByIds([1, 2, 3])
const [alice, bob, carol] = await Promise.all([
  userLoader.load(1),
  userLoader.load(2),
  userLoader.load(3),
]);
```

```ts
// Cache management
userLoader.prime(4, { id: 4, name: 'Dave' });
userLoader.clear(1);
userLoader.clearAll();
```

## Claude Code Integration

`@winglet/data-loader` ships Claude Code assets — the `data-loader-skill` expert skill and its knowledge base covering batching, caching, and advanced patterns — under `docs/claude/`. They are published alongside the package so AI assistants can reason about N+1 mitigation and scheduler semantics with package-specific context.

Inject those assets into your local Claude Code environment with the bundled `claude-sync` CLI (engine: `@slats/claude-assets-sync`):

```bash
# Direct dependency — bin is hoisted to your project's .bin
npx claude-sync --scope=user         # inject into ~/.claude (user-level)
npx claude-sync --scope=project      # inject into the nearest existing .claude, walking up from cwd

# Preview / overwrite flags
npx claude-sync --scope=user --dry-run   # show the plan without writing
npx claude-sync --scope=user --force     # overwrite locally-modified files

# Transitive dependency — pin the package explicitly
npx -p @winglet/data-loader claude-sync --scope=user
```

The CLI is a thin stub: it reads `@winglet/data-loader`'s own `package.json`, resolves `claude.assetPath` (`docs/claude`), and copies the tree into the selected `.claude` directory with hash-based change detection (skip unchanged, warn on divergence, require `--force` to overwrite).
