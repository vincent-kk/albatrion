## 📚 기술 용어 번역 사전

### 핵심 개발 용어

| 영문 | 한글 | 사용 예시 | 비고 |

| refactoring | 리팩토링 | "코드 리팩토링을 통해 가독성 향상" | 기능은 유지하며 구조 개선 |

### 위험도 레벨

| 레벨 | 이모지 | 한글 | 영문 | 사용 시나리오 |

| Critical | 🔴 | 위험 | Critical | 즉시 수정 필요한 심각한 문제 |
| High | 🟠 | 높음 | High | 빠른 시일 내 수정 필요 |
| Medium | 🟡 | 보통 | Medium | 검토 및 개선 권장 |
| Low | 🟢 | 낮음 | Low | 선택적 개선 사항 |

### 사용 예시

```markdown
- **위험도**: 🔴 위험

- **위험도**: 🟡 보통
- **위험도**: 🟢 낮음
```

### 맥락별 표현

#### 보안 관련
- 🔴 위험: SQL 인젝션 취약점
- 🟠 높음: XSS 공격 가능성

- 🟢 낮음: 민감 정보 로깅

- 🔴 위험: 무한 루프 가능성
- 🟠 높음: O(n²) 알고리즘
- 🟡 보통: 불필요한 재렌더링
- 🟢 낮음: 작은 최적화 기회

#### 하위 호환성
- 🔴 위험: 브레이킹 체인지

## ✍️ 문장 스타일 가이드

### 기본 원칙

1. **존댓말 사용**
   - ✅ "테스트를 추가하세요"
   - ✅ "변경사항을 확인해주세요"
   - ❌ "테스트를 추가해라"

2. **명확한 주어**
   - ✅ "이 함수는 빈 배열을 반환합니다"
   - ❌ "빈 배열을 반환합니다" (주어 불명확)

3. **능동태 선호**
   - ✅ "ValidationError를 throw합니다"
   - ⚠️ "ValidationError가 throw됩니다" (수동태)

4. **간결성**
   - ✅ "타입 안전성을 향상시킵니다"
   - ❌ "타입 안전성을 더욱 더 크게 향상시키도록 합니다"

### 문장 패턴

#### 설명 문장
```markdown
- {주어}는 {동작}을 {목적}합니다
- {기능}을 통해 {결과}를 달성합니다
- {변경사항}으로 {개선점}이 향상됩니다
```

**예시**:
- "이 함수는 사용자 입력을 검증합니다"
- "새로운 알고리즘을 통해 성능을 개선합니다"
- "타입 명시로 타입 안전성이 향상됩니다"

#### 권장 문장
```markdown
- {액션}을 {방법}으로 {목적}하세요
- {문제}를 해결하기 위해 {해결책}을 적용하세요
```

**예시**:
- "테스트를 추가하여 코드 품질을 보장하세요"
- "XSS 공격을 방지하기 위해 입력값을 새니타이즈하세요"

#### 경고 문장
```markdown
- {문제}가 발생할 수 있습니다
- {조건} 시 {위험}이 있습니다
- {상황}에 주의가 필요합니다
```

**예시**:
- "메모리 누수가 발생할 수 있습니다"
- "타입 검증 누락 시 런타임 에러가 발생할 수 있습니다"
- "브레이킹 체인지로 인해 하위 호환성에 주의가 필요합니다"

## 📊 통계 표현

### 숫자 표현

```markdown
**변경 통계**:
- 변경된 파일: {count}개
- 추가된 라인: +{count}
- 삭제된 라인: -{count}
- 순 변경: {+/-}{count}
```

### 백분율 표현

```markdown
- 테스트 커버리지: {percentage}%
- 성능 개선: {percentage}% 향상
- 번들 크기: {percentage}% 감소
```

### 비교 표현

```markdown
- 현재: {value_1} → 목표: {value_2}
- 이전: {value_1} → 현재: {value_2}
- {metric}: {old_value} → {new_value}
```

**예시**:
- 현재: 65% → 목표: 80%
- 이전: 1.2MB → 현재: 850KB
- 응답 시간: 500ms → 200ms

## ✅ 체크리스트: 보고서 작성 전 확인사항

### 용어 일관성
- [ ] 모든 기술 용어가 사전 정의대로 번역되었는가?
- [ ] 동일 용어를 일관되게 사용했는가?

### 위험도 표현
- [ ] 위험도 이모지가 올바르게 사용되었는가?
- [ ] 위험도 레벨이 명확히 표시되었는가?

### 문장 스타일
- [ ] 존댓말을 사용했는가?
- [ ] 명확한 주어가 있는가?
- [ ] 간결하고 명확한 문장인가?

### 띄어쓰기 & 문장 부호
- [ ] 영문과 한글 사이 띄어쓰기가 올바른가?
- [ ] 괄호 앞뒤 띄어쓰기가 올바른가?
- [ ] 마침표 사용이 일관되는가?

### 코드 블록
- [ ] 언어가 올바르게 지정되었는가?
- [ ] Before/After가 명확히 구분되는가?

### 전반적 품질
- [ ] 한글 독자가 이해하기 쉬운가?
- [ ] 전문성과 친근함의 균형이 있는가?
- [ ] 액션 아이템이 명확한가?

---

> **Best Practice**: 일관성 > 완벽성. 번역이 애매할 경우 원어를 병기하고, 프로젝트 전체에서 동일하게 사용하세요.
