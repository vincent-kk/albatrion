# ToT 평가 기준

## 평가 프레임워크 개요

총 100점 만점으로 5가지 축을 평가합니다.

## 1. 구현 복잡도 (30점)

**정의**: 해당 해석을 실제 코드로 구현할 때의 난이도

### 점수 기준표

| 점수 | 복잡도 수준 | 설명 | 예시 |
|------|-------------|------|------|
| **30점** | 매우 단순 | HTML/CSS만으로 구현 가능, 외부 라이브러리 불필요 | 단순 체크박스, 기본 input 필드 |
| **27점** | 단순 | 기본 React 컴포넌트, 간단한 상태 관리 (useState) | 토글 버튼, 드롭다운 메뉴 |
| **24점** | 보통-하 | 표준 라이브러리 1개, 중간 상태 관리 | Ant Design 컴포넌트 활용 |
| **20점** | 보통 | 외부 라이브러리 1-2개, useEffect/useContext 활용 | React DnD, 날짜 피커 |
| **17점** | 보통-상 | 여러 라이브러리 통합, 복잡한 훅 조합 | 차트 + 필터 + 실시간 업데이트 |
| **14점** | 복잡 | 커스텀 로직 많음, 상태 머신 필요 | 다단계 폼 위저드, 복잡한 검증 |
| **10점** | 매우 복잡 | 커스텀 엔진, 고급 알고리즘, 성능 최적화 필수 | 가상 스크롤, WebGL 렌더링 |

### 평가 시 고려사항

- **의존성 개수**: 라이브러리가 많을수록 점수 낮음
- **커스텀 로직 비율**: 기존 솔루션 활용 vs 직접 구현
- **상태 관리 복잡도**: useState vs Context vs Redux
- **비동기 처리**: Promise, async/await 사용 여부
- **에러 처리**: 엣지 케이스 처리 복잡도

**점수 감점 요인**:
- 외부 라이브러리 1개마다 -2점
- 커스텀 훅 필요 시 -2점
- 성능 최적화 필수 시 -3점
- 크로스 브라우저 이슈 예상 시 -2점

## 2. 요구사항 충족도 (30점)

**정의**: 사용자의 명시적/암묵적 니즈를 얼마나 충족하는가

### 점수 기준표

| 점수 | 충족 수준 | 설명 | 판단 기준 |
|------|-----------|------|-----------|
| **30점** | 완벽+ | 명시적 요구사항 + 예상 니즈까지 모두 충족 | 사용자가 요청하지 않은 편의 기능까지 제공 |
| **27점** | 완벽 | 모든 명시적 요구사항 100% 충족 | 기획서의 모든 항목 구현 |
| **24점** | 우수 | 핵심 요구사항 100% + 부가 요구사항 80% | 주요 기능 완벽, 일부 선택 기능 누락 가능 |
| **20점** | 양호 | 핵심 요구사항 100%, 부가 요구사항 일부 | 기본 사용 사례는 모두 커버 |
| **17점** | 보통 | 핵심 요구사항 80% | 일부 주요 기능 제한적 구현 |
| **14점** | 미흡 | 핵심 요구사항 60% | 기본 기능만 동작, 많은 제약 |
| **10점** | 최소 | 최소한의 기능만 제공 | MVP 수준 |

### 평가 매트릭스

**요구사항 분류**:
1. **필수 요구사항** (Must-have): 없으면 기능이 무의미
2. **중요 요구사항** (Should-have): 사용성에 큰 영향
3. **선택 요구사항** (Could-have): 있으면 좋음
4. **예상 니즈** (Would-be-nice): 사용자가 요청하지 않았지만 필요할 것

**점수 계산**:
```
점수 = (필수 충족률 × 0.5 + 중요 충족률 × 0.3 + 선택 충족률 × 0.15 + 예상 니즈 × 0.05) × 30
```

**예시**:
```
필수 4개 중 4개 충족 (100%)
중요 3개 중 2개 충족 (67%)
선택 2개 중 1개 충족 (50%)
예상 니즈 0개 충족 (0%)

점수 = (1.0×0.5 + 0.67×0.3 + 0.5×0.15 + 0×0.05) × 30
     = (0.5 + 0.201 + 0.075 + 0) × 30
     = 0.776 × 30
     = 23.3점 → 24점
```

## 3. UX 품질 (20점)

**정의**: 사용자 경험의 우수성 (직관성, 효율성, 접근성)

### 점수 기준표

| 점수 | UX 수준 | 설명 | 특징 |
|------|---------|------|------|
| **20점** | 탁월 | 직관적 + 효율적 + 접근성 우수 + 즐거움 | 사용자가 학습 없이 즉시 사용 가능 |
| **18점** | 우수 | 직관적 + 효율적 + 접근성 양호 | 1-2번 사용 후 완전 숙달 |
| **16점** | 양호 | 직관적이지만 효율성 보통 | 이해는 쉬우나 반복 작업 시 불편 |
| **14점** | 보통 | 학습 가능, 효율성 보통 | 튜토리얼이나 가이드 필요 |
| **12점** | 미흡 | 직관적이지 않지만 사용 가능 | 사용자가 시행착오 겪음 |
| **10점** | 불량 | 매우 불편, 자주 실수 유발 | 사용자 불만 예상 |
| **8점** | 매우 불량 | 사용 거의 불가능 | 대부분 사용자가 포기 |

### 평가 차원

#### 직관성 (Intuitiveness)
- [ ] 기능이 명확히 보이는가?
- [ ] 사용자가 다음 행동을 예측할 수 있는가?
- [ ] 일반적인 UX 패턴을 따르는가?
- [ ] 시각적 피드백이 즉각적인가?

#### 효율성 (Efficiency)
- [ ] 최소한의 클릭/입력으로 목표 달성 가능한가?
- [ ] 반복 작업에 최적화되어 있는가?
- [ ] 키보드 단축키 지원하는가?
- [ ] 자동완성/제안 기능이 있는가?

#### 접근성 (Accessibility)
- [ ] 키보드만으로 모든 기능 사용 가능한가?
- [ ] 스크린 리더 호환되는가?
- [ ] 색상 대비가 WCAG 2.1 AA 준수하는가?
- [ ] 모바일 터치 타겟 크기 적절한가? (44×44px 이상)

#### 즐거움 (Delight)
- [ ] 애니메이션이 자연스러운가?
- [ ] 마이크로 인터랙션이 있는가?
- [ ] 사용자에게 긍정적 감정을 주는가?

## 4. 유지보수성 (10점)

**정의**: 향후 코드 변경, 확장, 버그 수정의 용이성

### 점수 기준표

| 점수 | 유지보수성 | 설명 | 코드 특성 |
|------|-----------|------|-----------|
| **10점** | 매우 우수 | 단순 구조, 쉬운 변경 | 함수형, 작은 컴포넌트, 명확한 책임 |
| **9점** | 우수 | 표준 패턴, 적절한 추상화 | 일관된 코딩 스타일, 문서화 양호 |
| **8점** | 양호 | 일반적 구조, 변경 가능 | 일부 중복 코드, 문서화 부족 |
| **7점** | 보통 | 중간 복잡도, 주의 필요 | 상태 관리 복잡, 의존성 많음 |
| **6점** | 미흡 | 복잡한 구조, 변경 어려움 | 강한 결합, 테스트 어려움 |
| **4점** | 불량 | 매우 복잡, 레거시화 우려 | 스파게티 코드, 사이드 이펙트 많음 |
| **2점** | 매우 불량 | 거의 불가능, 재작성 필요 | 해독 불가능, 의존성 순환 |

### 평가 체크리스트

#### 코드 구조
- [ ] 단일 책임 원칙 준수하는가?
- [ ] 함수/컴포넌트 크기가 적절한가? (<100줄)
- [ ] 중복 코드가 없는가?
- [ ] 순환 의존성이 없는가?

#### 추상화 수준
- [ ] 적절한 추상화 레벨인가? (과도한 추상화 X)
- [ ] 재사용 가능한 유틸리티로 분리했는가?
- [ ] 인터페이스가 명확한가?

#### 테스트 가능성
- [ ] 순수 함수 비율이 높은가?
- [ ] 외부 의존성을 주입 가능한가?
- [ ] 모킹이 쉬운가?

#### 문서화
- [ ] 복잡한 로직에 주석이 있는가?
- [ ] 타입 정의가 명확한가?
- [ ] README가 있는가?

## 5. 팀 역량 적합성 (10점)

**정의**: 현재 팀의 기술 스택, 경험, 학습 능력과의 정합성

### 점수 기준표

| 점수 | 적합성 | 팀 상태 | 학습 기간 |
|------|--------|---------|-----------|
| **10점** | 완벽 | 팀이 해당 기술 완벽 숙련 | 즉시 가능 |
| **9점** | 우수 | 팀의 50% 이상이 경험 보유 | 1-3일 |
| **8점** | 양호 | 팀의 일부가 경험 보유 | 1주 |
| **7점** | 보통 | 유사 기술 경험 있음 | 2주 |
| **6점** | 학습 가능 | 학습 의지 있음, 자료 풍부 | 1개월 |
| **4점** | 학습 어려움 | 생소한 기술, 자료 부족 | 2-3개월 |
| **2점** | 팀 역량 외 | 전문가 필요, 외부 교육 필수 | 6개월+ |

### 평가 요소

#### 기술 스택 정합성
```yaml
프로젝트 기술: React + TypeScript + Jotai
해석 A: React DnD (라이브러리)
  - React 호환: ✅ 10점
  - TypeScript 지원: ✅ 10점
  - Jotai 통합: ✅ 10점
  총점: 10/10

해석 B: Vue Draggable (라이브러리)
  - React 호환: ❌ 0점 (프레임워크 불일치)
  총점: 0/10
```

#### 팀 경험 매트릭스
```
팀원 1: React DnD 프로젝트 경험 (3년)
팀원 2: 드래그앤드롭 구현 경험 (1년)
팀원 3: React 경험만 (DnD 없음)
팀원 4: React 초보

경험 보유율: 2/4 = 50% → 9점
```

#### 학습 곡선 평가
- **즉시 가능**: 팀의 80% 이상이 숙련 → 10점
- **1주 이내**: 공식 문서만으로 학습 가능 → 8점
- **1개월 이내**: 튜토리얼, 예제 필요 → 6점
- **3개월 이상**: 전문 교육, 멘토링 필요 → 4점

## 점수 해석 가이드

### 총점 범위별 의미

| 총점 | 평가 | 의미 | 권장 액션 |
|------|------|------|-----------|
| **85-100점** | 확실함 (Definitely) | 강력 추천, 즉시 진행 가능 | 선택 후 바로 구현 시작 |
| **75-84점** | 매우 가능함 (Highly Feasible) | 안전한 선택, 리스크 낮음 | 추가 검증 후 진행 |
| **70-74점** | 가능함 (Feasible) | 실행 가능, 주의 필요 | 리스크 완화 계획 수립 |
| **60-69점** | 주의 필요 (Caution) | 리스크 존재, 신중 검토 | 대안 준비 필수 |
| **50-59점** | 권장하지 않음 (Not Recommended) | 높은 리스크 | 다른 해석 고려 |
| **<50점** | 실행 불가 (Not Viable) | 실패 가능성 높음 | 절대 선택 금지 |

### 도메인별 가중치 조정

프로젝트 특성에 따라 평가 기준 가중치를 조정할 수 있습니다.

#### 스타트업/MVP
```yaml
구현 복잡도: 40점 (↑ +10)  # 빠른 출시 중요
요구사항 충족도: 25점 (↓ -5)  # 최소 기능만
UX 품질: 15점 (↓ -5)
유지보수성: 10점 (↓ 0)
팀 역량: 10점 (↓ 0)
```

#### 엔터프라이즈
```yaml
구현 복잡도: 20점 (↓ -10)  # 품질 우선
요구사항 충족도: 30점 (↓ 0)
UX 품질: 20점 (↓ 0)
유지보수성: 15점 (↑ +5)  # 장기 유지보수
팀 역량: 15점 (↑ +5)  # 팀 안정성
```

#### 사용자 중심 제품
```yaml
구현 복잡도: 25점 (↓ -5)
요구사항 충족도: 30점 (↓ 0)
UX 품질: 30점 (↑ +10)  # UX 최우선
유지보수성: 10점 (↓ 0)
팀 역량: 5점 (↓ -5)
```

## 평가 일관성 유지

### 동일 기준 적용
- 모든 후보를 **동일한 평가표**로 점수화
- 평가자 편향 최소화
- 객관적 증거 기반 점수 부여

### 상대 평가 금지
- 각 후보를 **절대 기준**으로 평가
- "A가 B보다 낫다"가 아닌 "A는 85점, B는 78점"

### 점수 정당화
- 모든 점수에 **구체적 근거** 제시
- "28/30 (매우 간단)" ← ✅ 좋음
- "28/30" ← ❌ 불충분

---

> **핵심 원칙**: 객관적 기준 기반 평가, 주관적 판단 최소화
> **품질 보증**: 모든 점수는 측정 가능하고 검증 가능해야 함
