# 해석 템플릿

## 해석 후보 작성 템플릿

### 기본 형식

```markdown
### 해석 {ID}: {간결한 제목}

**설명**: {구체적인 구현 방법 2-3줄}

**구현 요소**:
- UI: {사용할 컴포넌트/라이브러리}
- 상태 관리: {상태 관리 방식}
- 데이터 흐름: {데이터 처리 방법}
- 특이사항: {주의할 점, 제약사항}

**장점**:
- {장점 1}
- {장점 2}
- {장점 3}

**단점**:
- {단점 1}
- {단점 2}
```

### 예시

```markdown
### 해석 A: 단순 체크박스 목록

**설명**: 시간대별 체크박스를 세로로 나열하여 사용자가 클릭으로 선택하는 방식. Ant Design Mobile의 Checkbox 컴포넌트 활용.

**구현 요소**:
- UI: Ant Design Mobile Checkbox.Group
- 상태 관리: Jotai atom (selectedTimeSlots: string[])
- 데이터 흐름: 체크박스 onChange → atom 업데이트 → 저장 버튼 활성화
- 특이사항: 시간대 100개 이상 시 가상 스크롤 고려 필요

**장점**:
- 구현 매우 간단 (1-2시간)
- 표준 UI 패턴, 사용자 학습 불필요
- 모바일 터치 이벤트 완벽 지원
- 접근성 우수 (키보드, 스크린 리더)

**단점**:
- 많은 시간대 선택 시 불편 (일일이 클릭)
- 연속된 시간 범위 선택에 비효율적
- 시각적 직관성 떨어짐
```

## 평가 표 템플릿

### 마크다운 형식

```markdown
## 평가 결과

| 해석 | 구현 복잡도 (30) | 요구사항 충족도 (30) | UX 품질 (20) | 유지보수성 (10) | 팀 역량 (10) | **총점** | **평가** |
|------|------------------|---------------------|--------------|----------------|-------------|----------|----------|
| A    | 28               | 20                  | 12           | 9              | 10          | **79**   | 가능함   |
| B    | 20               | 28                  | 18           | 7              | 9           | **82**   | 확실함 ✅ |
| C    | 15               | 25                  | 17           | 6              | 5           | **68**   | 주의 필요 |
| D    | 27               | 18                  | 10           | 8              | 10          | **73**   | 가능함   |

### 상세 평가

**해석 A 평가**:
- 구현 복잡도: 28/30 (매우 간단 - 표준 컴포넌트만 사용)
- 요구사항 충족도: 20/30 (기본 기능만 제공, 범위 선택 어려움)
- UX 품질: 12/20 (직관적이지만 비효율적)
- 유지보수성: 9/10 (단순 구조, 변경 쉬움)
- 팀 역량: 10/10 (표준 React 지식만 필요)
- **총점: 79/100 (가능함)**

**해석 B 평가**:
- 구현 복잡도: 20/30 (중간 - react-dnd 라이브러리 필요)
- 요구사항 충족도: 28/30 (우수한 UX, 범위 선택 편리)
- UX 품질: 18/20 (직관적이고 효율적, 터치 이슈 가능)
- 유지보수성: 7/10 (DnD 상태 관리 복잡)
- 팀 역량: 9/10 (팀이 React DnD 경험 보유)
- **총점: 82/100 (확실함)** ✅ **선택**

**해석 C 평가**:
- 구현 복잡도: 15/30 (복잡 - 캘린더 라이브러리 + 커스텀 로직)
- 요구사항 충족도: 25/30 (좋은 UX, 시각적 우수)
- UX 품질: 17/20 (시각적으로 우수하나 학습 필요)
- 유지보수성: 6/10 (캘린더 통합 복잡)
- 팀 역량: 5/10 (캘린더 라이브러리 미경험)
- **총점: 68/100 (주의 필요)**

**해석 D 평가**:
- 구현 복잡도: 27/30 (간단 - 기본 input 필드)
- 요구사항 충족도: 18/30 (최소 기능, UX 불량)
- UX 품질: 10/20 (불편함, 실수 유발)
- 유지보수성: 8/10 (간단)
- 팀 역량: 10/10 (기본 HTML)
- **총점: 73/100 (가능함)**
```

## Lookahead 템플릿

### 간단한 형식
```markdown
**Lookahead (다음 의사결정 예측)**:
- 다음 결정: {구체적인 다음 선택 사항}
- 예상 복잡도: {낮음|중간|높음}
- 리스크: {예상 문제점 2-3개}
- 의존성: {필요한 선행 작업}
```

### 상세 형식
```markdown
**Lookahead (다음 의사결정 예측)**:

### 다음 결정 포인트
1. **드래그 라이브러리 선택**
   - 옵션: react-dnd vs react-beautiful-dnd vs @dnd-kit/core
   - 평가 기준: TypeScript 지원, 모바일 호환성, 성능
   - 예상 소요 시간: 반나절 (조사) + 1일 (PoC)

2. **모바일 대응 전략**
   - 옵션: 반응형 vs 별도 모바일 UI vs 하이브리드
   - 평가 기준: 개발 비용, UX 품질
   - 예상 소요 시간: 1-2일

### 예상 리스크
| 리스크 | 확률 | 영향도 | 완화 방안 |
|--------|------|--------|-----------|
| 모바일 터치 이벤트 미지원 | 중간 | 높음 | 초기 모바일 테스트 우선 |
| 성능 문제 (100개+ 시간대) | 낮음 | 중간 | 가상 스크롤 적용 |
| 접근성 이슈 | 중간 | 중간 | 키보드 핸들러 추가 |

### 필요한 의존성
- React 18+ (이미 충족 ✅)
- TypeScript strict mode (설정 필요 ⚠️)
- Jotai atoms 구조 설계 (미완성 ❌)
```

## Backtrack Plan 템플릿

### 간단한 형식
```markdown
**Backtrack Plan (대안 계획)**:
- 대안: {2순위 해석 ID 및 이름}
- 트리거 조건: {언제 전환할 것인가}
- 예상 손실: {시간/비용}
- 복구 시간: {대안으로 전환 소요 시간}
```

### 상세 형식
```markdown
**Backtrack Plan (대안 계획)**:

### 대안 해석
- **ID**: A
- **제목**: 단순 체크박스 목록
- **점수**: 79/100 (가능함)
- **장점 (vs 선택된 B)**:
  * 구현 단순 (1-2일)
  * 모바일 이슈 없음
  * 접근성 완벽

### 트리거 조건 (하나라도 해당 시 전환)
1. **일정 트리거**:
   - 구현 시작 후 2일 경과했으나 기본 드래그 동작 미완성
   - 출시 1주일 전까지 미완성

2. **기술 트리거**:
   - 모바일 터치 이벤트 이슈로 1일 이상 블로킹
   - 성능 목표 미달 (첫 렌더링 >500ms, 인터랙션 >100ms)
   - 접근성 WCAG 2.1 AA 기준 충족 불가

3. **리소스 트리거**:
   - 팀원 부재로 React DnD 지원 불가
   - 우선순위 변경으로 개발 시간 1일로 단축

### 전환 비용 및 복구
| 항목 | 상세 |
|------|------|
| **시간 손실** | 이미 투입한 2-3일 |
| **재사용 가능 코드** | Jotai atom, 타임슬롯 데이터 구조, 저장 API 호출 로직 |
| **폐기 코드** | DnD 관련 컴포넌트 및 이벤트 핸들러 |
| **복구 시간** | 1일 (체크박스 UI는 매우 단순) |
| **총 지연** | 2-3일 (손실) - 재사용 코드 덕분에 완전 재작업은 아님 |

### 의사결정 기준
전환을 결정할 때 고려할 우선순위:
1. **일정 압박** (출시 2주 이내) → 즉시 전환
2. **기술 난이도** (해결책 보이지 않음) → 1일 더 시도 후 전환
3. **품질 기준** (성능/접근성 미달) → 완화 가능 여부 검토 후 결정
```
